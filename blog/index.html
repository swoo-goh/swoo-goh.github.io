      #Analytics Engineering#Cloud Engineering#Data Engineering

      1. Intro: ‘약속된 낙원’은 없었다 (The Cloud Paradox) 수년 전, 우리가 안정적인 온프레미스(On-Premise) 환경을 뒤로하고 대대적인 ‘클라우드 마이그레이션’을 감행했던 이유는 명확했습니다. 첫째는 **인프라 비용의 획기적 절감(Cost Efficiency)**이었고, 둘째는 하드웨어 관리에서 해방되어 **전문적인 서비스 관리(Managed Service)**를 받기 위함이었습니다. 하지만 2025년 현재, 우리가 받아든 성적표는 어떻습니까? **‘비용 절감’**은커녕 예측 불가능한 변동비 폭탄을 맞고 있으며, **‘관리의 편의성’**은 복잡해진 아키텍처와 청구서 분석이라는 새로운 관리 비용으로 치환되었습니다. 우리는 ‘하드웨어’라는 짐을 내려놓은 대신, ‘비용’이라는 더 무거운 짐을 지게 되었습니다.

      클라우드 예산의 32%가 사라지고 있습니다: 인프라가 아닌 '쿼리'를 수술해야 할 때

      #Analytics Engineering#Cloud Engineering#Data Engineering

      “시스템은 거짓말을 하지 않습니다. 다만 청구서로 비명을 지를 뿐입니다.” 클라우드 비용 절감(FinOps)을 논할 때, 대부분의 관리자는 이렇게 외칩니다. “주말에 개발 서버 껐어? 스팟 인스턴스(Spot Instance) 적용했어?” 물론 중요합니다. 하지만 이것은 수도꼭지 잠그기에 불과합니다. 진짜 문제는 수도꼭지가 아니라, 파이프라인 안에서 줄줄 새고 있는 물(Resource) 그 자체에 있습니다. 오늘은 인프라 엔지니어가 아닌, 데이터 엔지니어/아키텍트의 관점에서 비용 낭비의 진짜 원인을 진단해 봅니다. 1. 팩트 체크 : 낭비되는 돈의 실체 Flexera의 2024 State of the Cloud Report에 따르면, 기업들은 클라우드 예산의 최소 **32%**가 낭비되고 있다고 스스로 인정했습니다. 재미있는 사실은, 실제 낭비 규모는 이보다 훨씬 클 것이라는 점입니다.

      OS 설치 및 부트 로더(GRUB) 설정커널이 하드웨어를 제어하는 방식부터 통제해야 합니다.Distro: Ubuntu 24.04 LTS 또는 RHEL 9.x (Kernel 6.8+ 권장).GRUB Parameters (/etc/default/grub):intel_pstate=performance: CPU 클럭 변동을 막고 최대 성능 유지.processor.max_cstate=0: C-state 깊은 진입을 막아 Wake-up 레이턴시 제거.iommu=pt: RDMA 장치의 Direct Memory Access 시 성능 저하 방지 (Pass-through).hugepagesz=1G hugepages=32: 셔플 및 RDMA 버퍼용 거대 페이지 사전 할당 (TLB Miss 방지).transparent_hugepage=never: 커널의 자동 HugePage 할당 시 발생하는 스파이크(Jitter) 원천 차단.2. NVMe 스토리지 최적화 (6개월 전문가용)SSD가 가진 IOPS를 끝까지 뽑아내기 위한 설정입니다.파일 시스템: XFS (메타데이터 병렬 처리에 최적).Mount 옵션 (/etc/fstab):Bash/dev/nvme0n1p1 /mnt/data xfs defaults,noatime,nodiratime,logbsize=256k,logbufs=8 0 0 noatime, nodiratime: 읽기 시마다 발생하는 쓰기(Access Time 기록) 낭비 제거.logbsize, logbufs: 로그 버퍼를 키워 작은 쓰기 빈도 감소.I/O Scheduler: * echo none > /sys/block/nvme0n1/queue/scheduler (NVMe 내부 큐에 전적으로 맡김).3. 커널 파라미터 튜닝 (/etc/sysctl.conf)데이터 이동 효율을 극대화하고 커널의 간섭을 최소화합니다.분류파라미터권장 값분석적 의미Memoryvm.swappiness0어떤 경우에도 SSD를 스왑(Swap)으로 쓰지 않음.Memoryvm.dirty_ratio40OS 쓰기 버퍼를 키워 SSD로의 Flush 지연.Memorykernel.numa_balancing0NUMA 노드 간 자동 이동 방지 (Jitter 제거).Networknet.core.rmem_max16777216RDMA Fallback(TCP) 시 대용량 버퍼 확보.Networknet.core.netdev_max_backlog30000고속 네트워크 패킷 유실 방지.4. RDMA & 고속 네트워크 준비2주 뒤 카드가 왔을 때 즉시 인식하기 위한 환경입니다.Interrupt Affinity: * irqbalance 서비스 중지.네트워크 카드의 인터럽트를 특정 CPU 코어에 수동 바인딩(set_irq_affinity.sh).Locked Memory Limit:/etc/security/limits.conf에서 memlock을 unlimited로 설정. (RDMA Memory Registration 필수 조건)⚠️ 리스크 기반의 직설적 제언: “깨끗한 도화지가 아니면 분석은 오염됩니다"Bloatware의 저주: 설치 시 ‘Standard System Utilities’ 외에는 아무것도 깔지 마십시오. 특히 Snap, Docker, Firewall(ufw/firewalld) 등은 당신이 모르는 사이에 백그라운드에서 CPU 사이클을 훔쳐가며 perf 수치를 망가뜨릴 것입니다.Thermal Throttling: OS 설치 후 반드시 하드웨어 모니터링 툴로 Idle 온도를 체크하십시오. 6개월 SSD 전문가라면 아시겠지만, 쿨링 솔루션이 부실하면 RDMA 전송 시 발생하는 열로 인해 SSD가 스스로 속도를 늦추는(Throttling) 현상이 발생합니다. 이건 소프트웨어로 못 고칩니다.전문가적 조언: 파트너님, OS를 깔고 나서 가장 먼저 해야 할 일은 **‘시스템 침묵 측정’**입니다. 아무것도 안 할 때 IPC가 0.01 이하로 떨어지는지 확인하십시오. 여기서 노이즈가 잡히지 않으면, 2주 뒤의 데이터는 모두 가짜입니다.
