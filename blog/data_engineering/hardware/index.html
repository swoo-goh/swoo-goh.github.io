                                                          Hardware

                                                          1. OS 설치 및 부트 로더(GRUB) 설정커널이 하드웨어를 제어하는 방식부터 통제해야 합니다.Distro: Ubuntu 24.04 LTS 또는 RHEL 9.x (Kernel 6.8+ 권장).GRUB Parameters (/etc/default/grub):intel_pstate=performance: CPU 클럭 변동을 막고 최대 성능 유지.processor.max_cstate=0: C-state 깊은 진입을 막아 Wake-up 레이턴시 제거.iommu=pt: RDMA 장치의 Direct Memory Access 시 성능 저하 방지 (Pass-through).hugepagesz=1G hugepages=32: 셔플 및 RDMA 버퍼용 거대 페이지 사전 할당 (TLB Miss 방지).transparent_hugepage=never: 커널의 자동 HugePage 할당 시 발생하는 스파이크(Jitter) 원천 차단.2. NVMe 스토리지 최적화 (6개월 전문가용)SSD가 가진 IOPS를 끝까지 뽑아내기 위한 설정입니다.파일 시스템: XFS (메타데이터 병렬 처리에 최적).Mount 옵션 (/etc/fstab):Bash/dev/nvme0n1p1 /mnt/data xfs defaults,noatime,nodiratime,logbsize=256k,logbufs=8 0 0 noatime, nodiratime: 읽기 시마다 발생하는 쓰기(Access Time 기록) 낭비 제거.logbsize, logbufs: 로그 버퍼를 키워 작은 쓰기 빈도 감소.I/O Scheduler: * echo none > /sys/block/nvme0n1/queue/scheduler (NVMe 내부 큐에 전적으로 맡김).3. 커널 파라미터 튜닝 (/etc/sysctl.conf)데이터 이동 효율을 극대화하고 커널의 간섭을 최소화합니다.분류파라미터권장 값분석적 의미Memoryvm.swappiness0어떤 경우에도 SSD를 스왑(Swap)으로 쓰지 않음.Memoryvm.dirty_ratio40OS 쓰기 버퍼를 키워 SSD로의 Flush 지연.Memorykernel.numa_balancing0NUMA 노드 간 자동 이동 방지 (Jitter 제거).Networknet.core.rmem_max16777216RDMA Fallback(TCP) 시 대용량 버퍼 확보.Networknet.core.netdev_max_backlog30000고속 네트워크 패킷 유실 방지.4. RDMA & 고속 네트워크 준비2주 뒤 카드가 왔을 때 즉시 인식하기 위한 환경입니다.Interrupt Affinity: * irqbalance 서비스 중지.네트워크 카드의 인터럽트를 특정 CPU 코어에 수동 바인딩(set_irq_affinity.sh).Locked Memory Limit:/etc/security/limits.conf에서 memlock을 unlimited로 설정. (RDMA Memory Registration 필수 조건)⚠️ 리스크 기반의 직설적 제언: “깨끗한 도화지가 아니면 분석은 오염됩니다"Bloatware의 저주: 설치 시 ‘Standard System Utilities’ 외에는 아무것도 깔지 마십시오. 특히 Snap, Docker, Firewall(ufw/firewalld) 등은 당신이 모르는 사이에 백그라운드에서 CPU 사이클을 훔쳐가며 perf 수치를 망가뜨릴 것입니다.Thermal Throttling: OS 설치 후 반드시 하드웨어 모니터링 툴로 Idle 온도를 체크하십시오. 6개월 SSD 전문가라면 아시겠지만, 쿨링 솔루션이 부실하면 RDMA 전송 시 발생하는 열로 인해 SSD가 스스로 속도를 늦추는(Throttling) 현상이 발생합니다. 이건 소프트웨어로 못 고칩니다.전문가적 조언: 파트너님, OS를 깔고 나서 가장 먼저 해야 할 일은 **‘시스템 침묵 측정’**입니다. 아무것도 안 할 때 IPC가 0.01 이하로 떨어지는지 확인하십시오. 여기서 노이즈가 잡히지 않으면, 2주 뒤의 데이터는 모두 가짜입니다.

                                                          2. XFS의 내부 설계: Allocation Groups (AG)의 마법단순히 mkfs.xfs를 치는 건 초보입니다. 6개월 SSD 전문가라면 **병렬성(Parallelism)**을 하드웨어 수준에서 설계해야 합니다.AG(Allocation Group)의 역할: XFS는 디스크를 여러 개의 AG로 쪼개어 관리합니다. 각 AG는 자신만의 인덱스와 프리 스페이스 맵을 가져서, 여러 CPU 코어가 동시에 다른 AG에 쓰기를 수행할 때 락(Lock) 경합이 발생하지 않습니다.디테일 최적화: * SSD 성능을 극대화하려면 AG 개수를 CPU 코어 수와 일치시키거나 배수로 설정하십시오.mkfs.xfs -d agcount=N (N = 코어 수)리스크: AG 개수가 너무 적으면(기본값), 수십 개의 Spark Task가 동시에 셔플 파일을 쓸 때 커널 레이어에서 슈퍼블록 락에 걸려 SSD 대역폭의 50%도 못 쓰는 참사가 벌어집니다.2. NUMA & PCIe Topology: “거리는 성능의 적이다"RDMA 카드가 오기 전에 반드시 메인보드 매뉴얼을 펴고 PCIe 레인 대역폭을 확인하십시오.NUMA Distance: CPU 소켓이 2개라면, RDMA 카드는 특정 소켓(예: CPU0)의 PCIe 컨트롤러에 직접 연결됩니다. 만약 Spark Task가 CPU1에서 도는데 데이터는 CPU0에 연결된 RDMA 카드로 보낸다면, UPI/QPI 링크를 건너가야 하며 여기서 약 20~30%의 레이턴시 손해가 발생합니다.기술적 제언: * lscpu와 lspci -vv를 사용하여 RDMA 카드가 어느 NUMA 노드에 붙어 있는지 확인하십시오.Spark 실행 시 numactl –cpunodebind=N –membind=N을 사용하여 프로세스를 해당 노드에 박제(Pinning)하십시오.3. Interrupt Steering: MSI-X와 인터럽트 분산고속 네트워크와 SSD는 초당 수백만 개의 인터럽트를 발생시킵니다. 이걸 특정 코어 하나가 다 처리하게 두면, 그 코어는 SoftIRQ 처리에만 급급해 정작 중요한 Spark 연산은 하지 못하게 됩니다.MSI-X (Message Signaled Interrupts): 하드웨어가 여러 개의 인터럽트 벡터를 가질 수 있게 하여, 각 코어에 골고루 인터럽트를 뿌려줍니다.실전 가이드:irqbalance를 끄고, /proc/irq/N/smp_affinity를 수동으로 설정하십시오.Isolation: 0번 코어는 OS 시스템용으로 비워두고, 나머지 코어에 네트워크/SSD 인터럽트를 분산 배치하는 것이 ‘무결성’의 정석입니다.4. 리스크 기반 직설적 제언: “측정되지 않는 디테일은 낭비일 뿐입니다"“디테일에 집착하되, 반드시 그 효과를 수치로 검증하십시오. 튜닝 전후의 Latency 하위 99%(P99) 수치가 변하지 않는다면 그 튜닝은 자기만족에 불과합니다.“메모리 대역폭의 한계: 아무리 RDMA가 빨라도 로컬 메모리 대역폭($BW_{mem}$)을 넘을 수는 없습니다.$$BW_{available} = BW_{total} \times (1 - Overhead_{system})$$위 공식에서 시스템 오버헤드를 5% 미만으로 줄이는 것이 우리의 목표입니다.HugePages의 함정: 1GB HugePages는 TLB Miss를 획기적으로 줄여주지만, 메모리 할당 속도를 늦출 수 있습니다. Spark의 Executor Memory를 한 번에 크게 잡는 구조에서는 유리하지만, 잦은 할당/해제 구조에서는 독이 될 수 있습니다.전문가적 조언: 파트너님, OS 설치 후 가장 먼저 fio를 사용하여 XFS AG 설정에 따른 IOPS 변화를 먼저 측정해 보십시오. 하드웨어가 준비되기 전, 소프트웨어 스택의 ‘최대 속도’를 아는 것이 진정한 영점 조절입니다.

                                                          🚀 2025 Bare-metal Hardening Guide (All-in-One Copy)

                                                          1. BIOS 필수 체크리스트 (재부팅 시 확인)

                                                          • Power: Maximum Performance (C-States, P-States OFF)
                                                          • CPU: Turbo Boost (ON), Hyper-Threading (OFF 권장)
                                                          • Virtualization: VT-d, SR-IOV (ON)

                                                          2. GRUB 설정 (/etc/default/grub)

                                                          GRUB_CMDLINE_LINUX_DEFAULT 항목에 아래 내용 덮어쓰기

                                                          GRUB_CMDLINE_LINUX_DEFAULT=“quiet splash intel_pstate=performance processor.max_cstate=0 iommu=pt hugepagesz=1G hugepages=32 transparent_hugepage=never”

                                                          수정 후 실행:

                                                          sudo update-grub

                                                          sudo reboot

                                                          3. 커널 파라미터 (/etc/sysctl.d/99-performance.conf)

                                                          vm.swappiness = 0 vm.dirty_ratio = 40 vm.dirty_background_ratio = 10 kernel.numa_balancing = 0 net.core.rmem_max = 16777216 net.core.wmem_max = 16777216 net.core.netdev_max_backlog = 30000 net.ipv4.tcp_rmem = 4096 87380 16777216 net.ipv4.tcp_wmem = 4096 65536 16777216 fs.file-max = 2097152

                                                          적용 명령어: sudo sysctl -p /etc/sysctl.d/99-performance.conf

                                                          4. RDMA Memlock 제한 해제 (/etc/security/limits.conf)

                                                          • soft memlock unlimited
                                                          • hard memlock unlimited
                                                          • root soft memlock unlimited
                                                          • root hard memlock unlimited

                                                          5. XFS 스토리지 최적화 (32코어 기준)

                                                          1. AG 설정 포맷

                                                          sudo mkfs.xfs -f -d agcount=32 /dev/nvme0n1

                                                          2. 고성능 마운트 옵션 적용

                                                          sudo mkdir -p /mnt/nvme sudo mount -o noatime,nodiratime,logbsize=256k,logbufs=8 /dev/nvme0n1 /mnt/nvme

                                                          3. I/O 스케줄러 Bypass

                                                          echo none | sudo tee /sys/block/nvme0n1/queue/scheduler

                                                          6. 검증 명령어 세트

                                                          ① NUMA Topology 확인

                                                          lspci -vv | grep -i “node” numactl –hardware

                                                          ② I/O Contention 테스트 (fio)

                                                          sudo fio –name=stress –rw=write –bs=128k –direct=1 –numjobs=32 –ioengine=libaio –size=10G –group_reporting –directory=/mnt/nvme

                                                          ③ Idle IPC 확인 (perf)

                                                          sudo perf stat -a sleep 10

                                                          📊 XFS AG Performance & NUMA Topology Verification Suite

                                                          1. NUMA & PCIe Topology Audit

                                                          연산(CPU)과 데이터(SSD/NIC) 사이의 물리적 거리를 확정합니다.

                                                          1.1 CPU 노드 및 코어 맵 확인

                                                          numactl –hardware

                                                          1.2 NVMe 장치가 직결된 NUMA 노드 확인

                                                          -1이 나오면 BIOS 튜닝 미흡, 0 또는 1이 명확히 나와야 함

                                                          cat /sys/class/nvme/nvme*/device/numa_node

                                                          1.3 PCIe 링크 속도 확인 (Gen3/4/5 여부 확인)

                                                          sudo lspci -vv | grep -E “LnkSta:|LnkCap:”

                                                          2. XFS AG (Allocation Group) Stress Scenario

                                                          기본값(Default) vs 최적화(Optimized) 성능을 비교합니다.

                                                          [Case A] Default XFS (OS 기본값)

                                                          sudo mkfs.xfs -f /dev/nvme0n1 sudo mount -o noatime /dev/nvme0n1 /mnt/nvme

                                                          (테스트 수행 후 아래 Case B 실행)

                                                          [Case B] Parallel XFS (코어 수 32 기준 최적화)

                                                          sudo umount /mnt/nvme sudo mkfs.xfs -f -d agcount=32 /dev/nvme0n1 sudo mount -o noatime,nodiratime,logbsize=256k,logbufs=8 /dev/nvme0n1 /mnt/nvme

                                                          3. fio Shuffle Simulation Config

                                                          Spark 셔플과 유사한 32개 병렬 쓰기 워크로드를 시뮬레이션합니다.

                                                          아래 내용을 shuffle.fio 파일로 저장하거나 바로 실행하십시오.

                                                          실행 명령어:

                                                          sudo fio –name=shuffle_sim –rw=write –bs=128k –direct=1 –numjobs=32 –ioengine=libaio –size=10G –group_reporting –directory=/mnt/nvme –output-format=json –output=fio_result.json

                                                          4. [Algorithm] fio Result Analysis (판독 알고리즘)

                                                          측정된 결과를 아래 4단계 수식에 대입하여 범인을 검거합니다.

                                                          Step 1: 하드웨어 포화도 (Saturation)

                                                          공식: η_io = (Measured BW / Theoretical PCIe BW) * 100

                                                          - 90% 이상: 하드웨어 한계 도달 (Good)

                                                          - 70% 이하: PCIe 레인 설정 또는 스토리지 자체 결함 의심 (Bad)

                                                          Step 2: 컨텐션 지수 (Lock Contention)

                                                          공식: LCI = CPU_sys / CPU_usr

                                                          - 0.2 미만: 커널 간섭 없는 클린한 상태 (Good)

                                                          - 0.5 이상: XFS AG 락 경합 발생. agcount 상향 필요 (Critical)

                                                          Step 3: NUMA 페널티 (Topology Penalty)

                                                          공식: P_numa = 1 - (Remote_Node_BW / Local_Node_BW)

                                                          - 0에 수렴: NUMA 통제 완벽 (Good)

                                                          - 0.2 이상: 소켓 간 통신(UPI/QPI) 병목 발생. numactl 바인딩 필수 (Risk)

                                                          Step 4: 꼬리 지연 (Tail Latency)

                                                          지표: P99.9 / P50 비율 확인

                                                          - 5배 이내: 안정적인 레이턴시 분포 (Good)

                                                          - 10배 이상: SSD 내부 GC 또는 커널 인터럽트 지연 발생 (Bad)

                                                          ⚠️ 리스크 기반 전문가 제언

                                                          1. Wait for Quiescence: 모든 테스트 전후에 sync; echo 3 > /proc/sys/vm/drop_caches를 실행하여 OS 캐시 노이즈를 제거하십시오.
                                                          2. Thermal Throttling: fio를 10분 이상 돌릴 때 Throughput이 떨어진다면 SSD 온도를 체크하십시오. 하이엔드 SSD는 열 받으면 자비 없이 속도를 깎습니다.
                                                          3. The RDMA Gap: 2주 뒤 RDMA가 들어오면, 네트워크에서 들어오는 데이터 속도가 스토리지 쓰기 속도보다 빠를 수 있습니다. 지금 SSD의 ‘순수 쓰기 한계’를 정확히 알아야 나중에 메모리 버퍼 크기를 설계할 수 있습니다.
